# 项目任务钉钉规则与审核闭环设计

## 1. 目标

在项目创建阶段先配置项目成员，再创建任务；由项目设置默认钉钉通知规则，任务自动继承并可按需覆盖。通知只覆盖延期、任务变更审核和任务完成审核，不发送任务新建、派发或开始通知。

## 2. 范围

本次包含：

- 调整项目创建向导顺序；
- 项目级默认通知规则；
- 任务继承及可选覆盖；
- 钉钉接收人自动解析；
- 任务变更审核及项目经理执行；
- 任务完成提交及项目经理审核；
- 通知发送日志和失败信息。

本次不包含：

- 任务创建、派发、开始通知；
- T-3 到期提醒；
- 延期 3 天或 7 天升级；
- 部门负责人、技术总监或邮件通知；
- 新增在线审批引擎；现有审核能力继续使用。

## 3. 项目创建流程

创建向导调整为：

1. 选择模板；
2. 填写项目基本信息；
3. 配置项目成员；
4. 调整项目任务；
5. 配置项目默认通知规则；
6. 确认创建。

项目经理在选择后自动加入项目成员，且不能作为普通成员直接移除。其他成员必须在任务调整之前保存成功。任务负责人和协助人只能从当前项目的活跃成员中选择。

## 4. 通知规则

### 4.1 项目默认规则

项目保存以下默认事件：

| 事件 | 接收人 | 触发时点 |
|---|---|---|
| 任务延期 | 项目经理 | 任务首次进入延期状态时 |
| 任务变更提交 | 项目经理 | 变更申请成功进入待审核时 |
| 变更审核通过 | 任务负责人 | 项目经理审核通过后 |
| 变更审核驳回 | 任务负责人 | 项目经理审核驳回后 |
| 任务完成提交 | 项目经理 | 负责人提交完成并进入待审核时 |
| 完成审核通过 | 任务负责人 | 项目经理审核通过后 |
| 完成审核驳回 | 任务负责人 | 项目经理审核驳回后 |

项目默认不启用任务创建、派发和开始通知。

### 4.2 任务继承

任务默认使用项目规则。任务页展示：

- 使用项目默认规则；
- 自定义规则。

选择自定义时，只允许对上述事件启用或停用，不允许手工填写钉钉用户 ID。项目规则后续修改时，仍处于继承状态的任务即时使用新规则；已经自定义的任务保持自己的设置。

## 5. 接收人解析

业务页面只选择系统用户或项目角色，不直接维护钉钉 ID。发送时按以下顺序解析：

1. 从任务取得负责人或从项目取得项目经理；
2. 使用系统用户 ID 查询钉钉用户映射；
3. 取得有效钉钉用户标识；
4. 调用钉钉发送接口；
5. 保存发送结果。

孙文远在本轮验收数据中的系统用户标识为 1353。该值只能作为系统用户关系校验数据，不能作为项目经理日常手工输入项。

映射不存在或无效时不得改发给默认用户，也不得返回虚假的成功提示；应记录失败并显示明确原因。

## 6. 任务变更闭环

受控字段包括任务负责人、协助人、计划开始、计划结束、工期、任务内容、完成标准和输出物要求。

任务进入执行阶段后，受控字段不能通过“编辑任务”直接生效。流程为：

1. 用户发起任务变更，必须关联具体任务；
2. 记录变更前值、变更后值、原因和申请人；
3. 状态进入待审核，并通知项目经理；
4. 项目经理通过或驳回；
5. 驳回时不修改任务；
6. 通过后产生待执行状态；
7. 项目经理执行变更，系统在同一事务中更新任务并记录执行人和执行时间；
8. 将结果通知任务负责人。

变更编号由后端生成。所有列表和任务详情必须使用同一条变更记录，不得混入其他项目或任务的数据。

## 7. 任务完成闭环

1. 任务负责人提交完成；
2. 系统校验完成标准及既有输出物要求；
3. 任务进入待审核，通知项目经理；
4. 项目经理通过后任务进入已完成；
5. 项目经理驳回后任务恢复为进行中并保留真实进度；
6. 审核结果通知任务负责人。

任务不能只因进度达到 100% 自动变为已完成。

## 8. 通知日志

每次发送记录：

- 项目 ID、任务 ID；
- 业务事件；
- 系统接收人 ID、接收人姓名；
- 钉钉用户标识；
- 消息标题和摘要；
- 发送状态；
- 钉钉返回码或失败原因；
- 触发时间、发送时间和重试次数。

同一业务事件使用稳定的幂等键，避免重复提交或定时检查造成重复推送。

## 9. 验收标准

- 创建项目时，成员配置先于任务调整；
- 新增成员后无需刷新即可在任务负责人下拉中选择；
- 非项目成员不能被选为负责人或协助人；
